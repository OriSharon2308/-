# אזור הלמידה — מדריך המערכת

מדריך חי שמתעדכן בשלבים. מסביר את כל מה שנבנה לאזור הלמידה ואיך הכל
מתחבר למערכת הקיימת (Vela). אפשר לחזור אליו בכל שלב.

> **חשוב:** אזור הלמידה הוא *חלק אינטגרלי מאותה אפליקציה* — אותו מורה, אותו
> ילד, אותו זיכרון. לא מודול נפרד. הקוד מסודר, אבל הכל מחובר למערכת הקיימת.

---

## החזון בקצרה

באזור התרגול הקיים: מוצגת שאלה, והמורה עוזר לפי בקשה.
באזור הלמידה: המורה **מלמד** נושא מהבסיס, צעד אחר צעד, עד שליטה — עם **לוח
משותף** שעליו המורה מצייר כדי להסביר, והילד מצייר כדי לענות.

הבנייה בשלבים. סדר: **(א) זיכרון חכם → (ב) לוח → (ג) ממשק + בחירת נושאים →
(ד) הוראה בתוך נושא + בדיקת שליטה**. מסמך זה מכסה כרגע את שלב **א**.

---

## שלב א — זיכרון חכם משותף ✅

### הבעיה
היום, בכל פנייה למורה, נשלח אליו "ציר זמן" של אירועים גולמיים (כל תרגיל, כל
תשובה — `lib/memory.js`). ככל שהילד מתרגל, ההקשר תופח → יקר ואיטי.

### הפתרון: פרופיל ילד מתומצת ומובנה
במקום טקסט שמצטבר, מחזיקים **טבלה קטנה וקבועה בגודלה**: לכל נושא/תת-נושא
שורה אחת עם סטטוס והערה קצרה. זה כל מה שהמורה צריך כדי "לזכור את הילד",
והוא נשאר קטן לתמיד. **אותו פרופיל משמש את המורה בכל אזור** (תרגול + למידה).

**קובץ:** [`lib/learner-profile.js`](lib/learner-profile.js)
**אחסון:** `data/profiles/<userId>.json` — קובץ אחד לכל ילד.

```jsonc
{
  "userId": "u-...",
  "summary": "משפט חופשי על הילד (אופציונלי)",
  "topics": {
    "שעון":          { "status": "mastered",   "attempts": 6, "correct": 5, "streak": 3, "wrongStreak": 0, "note": "" },
    "חיבור תרגילים": { "status": "struggling",  "attempts": 4, "correct": 1, "streak": 0, "wrongStreak": 2, "note": "מתבלבל במעבר עשרת" }
  },
  "updatedAt": "ISO"
}
```

### סטטוסים
מפתח יציב באנגלית + תווית עברית לתצוגה:

| מפתח | עברית | מתי |
|------|--------|-----|
| `not_started` | לא התחיל | אין עדיין רשומה |
| `in_progress` | בתהליך | התחיל, עוד לא שולט ולא מתקשה |
| `struggling` | מתקשה | 2 טעויות ברצף, או ≥3 ניסיונות עם דיוק <50% |
| `mastered` | שולט | 3 נכונות ברצף, או ≥4 ניסיונות עם דיוק ≥80% |

הסטטוס מחושב **דטרמיניסטית בקוד** (`computeStatus`) — בלי AI: אמין 100%,
מיידי, וחינם. `"שולט"` דביק — לא יורד ממעידה אחת; חוזר ל`"מתקשה"` רק אחרי 2
טעויות ברצף (ככה המורה "לא מוותר", אבל גם מזהה הידרדרות אמיתית).
את הספים אפשר לכוונן ב-`computeStatus`.

### איך הפרופיל מתעדכן
אחרי כל אינטראקציה משמעותית, קוראים ל:

```js
learnerProfile.record(userId, { topic: "שעון", correct: true, note: "..." });
```

`record` סופר ניסיון/נכונה, מעדכן רצף/רצף-טעויות, ומחשב מחדש את הסטטוס.
**אין כתיבת AI ואין צבירת טקסט** — זה כל ה"זיכרון שמתעדכן".

המורה (AI) יכול גם לסמן וי ידנית כשהוא מחליט שהילד שולט:
```js
learnerProfile.setStatus(userId, "שעון", learnerProfile.STATUS.MASTERED, "הבין אחרי הסבר");
```

### ה-API של המודול
| פונקציה | תפקיד |
|---------|--------|
| `get(userId)` | הפרופיל המלא |
| `record(userId, {topic, correct, note})` | עדכון אחרי אינטראקציה (דטרמיניסטי) |
| `setStatus(userId, topic, status, note)` | קביעת סטטוס מפורשת (וי של המורה) |
| `setSummary(userId, text)` | משפט-סיכום חופשי (אופציונלי) |
| `weakTopics(userId)` | נושאים ב"מתקשה" — ל"המורה לא מוותר" בתרגול בהמשך |
| `toPromptText(userId)` | רינדור קצר בעברית להזרקה לפרומפט |
| `deleteUser(userId)` | מחיקה (למחיקת חשבון) |

## Prompt Caching — חיסכון בטוקנים ✅

**קובץ:** [`lib/llm.js`](lib/llm.js) (`buildSystemField`, `complete`).

### העיקרון
בכל פנייה למורה נשלח חלק **קבוע** (דמות המורה + שיטת ההוראה + הכלים +
פרופיל הילד) וחלק **משתנה** (הודעת הילד הנוכחית). את החלק הקבוע אפשר לשמור
ב-cache של Anthropic ולשלם עליו ~10% בלבד בפניות הבאות (וגם מהר יותר).

### איך זה מוגדר
שולחים את ה-`system` כמערך "בלוקים", ומסמנים על הבלוק שרוצים לשמור:
`cache_control: { type: "ephemeral" }`. Anthropic שומר את כל ה-prefix עד
הסימון. בפנייה הבאה, אם ה-prefix **זהה בתו** — פגיעת מטמון.

הקוד עוטף את זה: מעבירים `cacheSystem: true` (או `system` כמערך
`[{text, cache:true}]` לשליטה עדינה):

```js
await llm.complete({
  system: stableTeacherSystem,   // קבוע: דמות + שיטה + כלים + פרופיל
  cacheSystem: true,             // ← נשמר במטמון
  messages: [{ role: "user", content: childMessage }],  // משתנה — מחוץ למטמון
});
```

### דגשים
- **תנאי סף:** מטמון נכנס לפעולה רק כשה-prefix ≥ ~1024 טוקנים (Sonnet/Opus)
  או ~2048 (Haiku). קצר מדי → פשוט לא נשמר, בלי שגיאה. ה-system של מורה
  אזור-הלמידה (דמות + שיטה + כלים + פרופיל) ארוך מספיק.
- **יציבות ה-prefix:** המטמון עובד רק אם ה-prefix זהה בתו בין פניות. לכן כל
  מה שמשתנה כל הודעה (טקסט הילד) חייב להישאר ב-`messages`, לא ב-`system`.
- **חי ~5 דקות** ומתחדש בכל פגיעה. בלי כותרת beta — זה GA במודלים הנוכחיים.
- **לוג:** `complete` מדפיס `[cache] created=… read=…`. `read>0` = חיסכון.
- **תאימות לאחור:** בלי `cacheSystem`, `system` נשלח כמחרוזת רגילה — כל
  הקוד הקיים ממשיך לעבוד בדיוק כמו קודם.

## איך זה מחובר למערכת הקיימת ✅

נשמר העיקרון: **לא נגענו בלוגיקת המורה/הקול** — רק הרחבנו וחיברנו.

| קובץ | מה השתנה | למה |
|------|-----------|-----|
| [`lib/learner-profile.js`](lib/learner-profile.js) | **חדש** | הפרופיל המשותף |
| [`lib/llm.js`](lib/llm.js) | תמיכת caching (תואם לאחור) | חיסכון |
| [`agent/orchestrator.js`](agent/orchestrator.js) | מזריק `toPromptText` להקשר המורה + קורא `record` אחרי תשובה | הזיכרון המשותף נקרא ומתעדכן מהתרגול |
| [`server.js`](server.js) | `learnerProfile.deleteUser` במחיקת חשבון | היגיינת נתונים |

כל החיבורים ב-`orchestrator` עטופים ב-`try/catch` — אם משהו בפרופיל נכשל,
הצ'אט הקיים ממשיך לעבוד כרגיל. הזיכרון המובנה הוא **תוספת, לא תלות**.

> כשנבנה את מורה אזור-הלמידה (שלב ד), הוא יקרא ל-`llm.complete` עם
> `cacheSystem: true` ו-`toPromptText` בתוך ה-system — וכך ייהנה גם
> מהזיכרון המשותף וגם מה-caching.

```bash
node scripts/test-profile.js
```

מדגים: (1) מעברי סטטוס (לא התחיל → בתהליך → שולט / מתקשה, + וי ידני),

הבדיקה משתמשת ב-userId מדומה ומוחקת אותו בסוף.

## שלב ב — הלוח ✅

הלוח המשותף: עליו המורה מצייר כדי להסביר, והילד מצייר כדי לענות. נבנה
כרכיב עצמאי לשימוש חוזר — יוטמע בממשק אזור הלמידה בשלב ג.

### קבצים
| קובץ | תפקיד |
|------|--------|
| [`board.js`](board.js) | **רכיב הלוח** (`VelaBoard`) — סצנה, רקע, כלי מורה, ציור ילד |
| [`lib/board-tools.js`](lib/board-tools.js) | **סכמת הכלים הקנונית** (Anthropic Tool Use) — מקור אמת אחד |
| [`board-test.html`](board-test.html) | **דף בדיקה ידני** — `/board-test` (מאחורי התחברות) |

### מערכת הצירים (זהה ב-board.js וב-board-tools.js)
מרחב לוגי קבוע **800×560**, ראשית (0,0) שמאל-עליון, x ימינה, y מטה,
משבצת רשת = **40** (⇒ 20×14). הקואורדינטות יציבות — לא תלויות בגודל המסך
(הקנבס מותאם לתצוגה דרך DPR + CSS, אבל כל הציור במרחב הלוגי).

### רקע: רשת או נקודות
`board.setBackground('grid' | 'dots')` או `board.toggleBackground()`.
בדף הבדיקה — מתג "משבצות / נקודות".

### כלי המורה (Tool Use) — מוגדרים פעם אחת
מוגדרים ב-[`lib/board-tools.js`](lib/board-tools.js) כסכמת Anthropic
(`name` + `description` + `input_schema`). זה מה ש-Claude יקבל (ובהמשך
נשמור אותם ב-Prompt Cache יחד עם ה-system, כך שהמורה "לומד" אותם פעם אחת).

| כלי | פרמטרים | מה עושה |
|-----|----------|----------|
| `draw_circle` | `x, y, r, color?` | עיגול במרכז (x,y) |
| `draw_line` | `x1, y1, x2, y2, color?, width?` | קו ישר |
| `write_text` | `x, y, text, size?, color?` | טקסט/מספר ממורכז |
| `clear_board` | — | מנקה הכל |

**נקודת כניסה אחת בלקוח:** `board.tool(name, input)`. כש-Claude יחזיר
`tool_use` (שלב ד), הלקוח פשוט יקרא `board.tool(block.name, block.input)` —
בלי מיפוי נוסף. **הוספת כלי** = רשומה ב-board-tools.js + handler ב-`_tools`
שב-board.js (אותו שם). זה הכל.

### ציור הילד — נשמר כקואורדינטות (בלי ניתוח עדיין)
`board.setMode('child')` → ציור חופשי בעכבר/מגע (Pointer Events מאחדים
עכבר/מגע/עט). כל קו נשמר כ-`{ who:'child', type:'stroke', points:[{x,y}…] }`.
שליפה: `board.getChildStrokes()` / הסצנה המלאה: `board.getScene()`.
בשלב הבא ננתח את הקואורדינטות האלה כדי שהמורה "יבין" מה הילד צייר.

### בדיקה
פותחים `/board-test` (אחרי התחברות): מתג רקע, כפתורים לכל כלי מורה (עם
שדות x/y/r/טקסט), דוגמאות מהירות (ציר מספרים, עיגולים לספירה), מצב "ציור
ילד", ופאנל חי שמציג את הקואורדינטות שנשמרות ואת ה-scene המלא.
> הערה: רינדור הקנבס נבדק חזותית בדפדפן (אין headless כאן); השרת והקבצים
> נבדקו אוטומטית (תחביר, עקביות שמות כלים, הגשה מאחורי התחברות).

## המורה מסרטט על הלוח (Tool Use) ✅

המורה (Claude) מסרטט בפועל על הלוח לפי בקשה בצ'אט — "צייר לי משולש",
"סרטט ציר מספרים", "צייר שעון". סרטוט הסברתי ופונקציונלי, לא אמנותי.

### איך זה עובד
1. הילד כותב בקשה בצ'אט (בעמוד `/learn-board`).
2. הלקוח שולח ל-`POST /api/teach` את ההודעה + גודל הלוח (`geometry`) +
   תקציר אזורים תפוסים (`occupied`, כדי לא לסרטט מעל הקיים).
3. [`agent/draw-agent.js`](agent/draw-agent.js) בונה system קבוע (דמות +
   הוראות סרטוט + מערכת צירים) ושולח ל-Claude עם **כל כלי הלוח** כ-Tool Use
   ([`lib/board-tools.js`](lib/board-tools.js)), עם Prompt Caching.
4. **לולאת Tool Use**: לכל קריאת כלי מחזירים ל-Claude אישור (`tool_result`)
   כדי שישלים ציורים מרובי-שלבים (שעון = עיגול + 12 מספרים + מחוגים).
5. השרת מחזיר `{ reply, toolCalls }`; הלקוח מריץ `board.runTools(toolCalls)`
   והסרטוט מופיע על הלוח.

### כלי הסרטוט של המורה
[`lib/board-tools.js`](lib/board-tools.js) — הגדרה קנונית אחת, מודולרי
(הוספת כלי = רשומה כאן + handler באותו שם ב-[`board.js`](board.js)):

| כלי | מה מסרטט |
|-----|-----------|
| `draw_circle` | עיגול |
| `draw_line` | קו ישר |
| `draw_polygon` | מצולע סגור (משולש/ריבוע...), אופציה למילוי |
| `draw_arrow` | חץ עם ראש |
| `draw_point` | נקודה מסומנת + תווית |
| `draw_number_line` | ציר מספרים עם שנתות וערכים |
| `write_text` | טקסט/מספר |
| `clear_board` | ניקוי הלוח |

### תשתית
- [`lib/llm.js`](lib/llm.js) → `completeTools()` — קריאה עם `tools`, מחזיר
  `{ text, toolCalls, content, stopReason }`.
- **לא נגענו** במורה/קול/זיכרון הקיימים: זה מודול נפרד (`draw-agent`) +
  endpoint נפרד (`/api/teach`). אזור התרגול (`/api/chat`) לא הושפע.

### בדיקה
בעמוד `/learn-board`, לחיצה על המורה → צ'אט, ולכתוב:
"צייר לי משולש" · "סרטט ציר מספרים מ-0 עד 10" · "צייר שעון עם מחוגים על 3" ·
"צייר ריבוע אדום" · "סמן נקודה במרכז" · "נקה את הלוח".
